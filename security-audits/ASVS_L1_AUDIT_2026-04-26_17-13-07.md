### HIGH-03 — Sanitiseur HTML insuffisant

**Fichier :** `src/middlewares/validators.js:4-6`

La fonction actuelle `value.replace(/[<>]/g, "")` ne protège pas contre les payloads encodés (`&#60;`, `&#x3C;`, `javascript:`, etc.).

Installer `sanitize-html` :
```
npm install sanitize-html
```

Remplacer `sanitizeValue()` dans `src/middlewares/validators.js` :

```js
const sanitizeHtml = require("sanitize-html");

function sanitizeValue(value) {
  if (typeof value !== "string") return value;
  return sanitizeHtml(value, { allowedTags: [], allowedAttributes: {} }).trim();
}
```

`allowedTags: []` et `allowedAttributes: {}` stripent tous les tags HTML et leurs attributs — seul le texte brut passe. Cela couvre les encodages HTML, les attributs d'événements, et les data URIs. Note : le risque actuel est faible grâce à l'auto-escaping EJS (`<%= %>`), mais ce fix est la défense en profondeur correcte.

### MED-02 — Race condition sur le stock au checkout

**Fichier :** `src/services/orderService.js:73-84`

Le panier est chargé avant la transaction (`loadCart` ligne 73), donc la vérification de stock porte sur des données potentiellement périmées. Déplacer le chargement des items du panier à l'intérieur de la transaction avec un verrou `LOCK.UPDATE` :

```js
const order = await sequelize.transaction(async (transaction) => {
  // Recharger les items avec verrou à l'intérieur de la transaction
  const lockedItems = await models.CartItem.findAll({
    where: { cartId: cart.id, savedForLater: false },
    include: [{ model: models.Product, as: "product" }],
    lock: transaction.LOCK.UPDATE,
    transaction
  });

  for (const item of lockedItems) {
    if (item.product.stock < item.qty)
      throw new AppError(`Stock insuffisant pour ${item.product.name}`, 400, "OUT_OF_STOCK");
  }
  // ... reste de la transaction identique, utiliser lockedItems à la place de items
});
```

Le `LOCK.UPDATE` garantit qu'aucune autre transaction parallèle ne peut lire ou modifier le stock de ces produits entre la vérification et le décrément.

### MED-06 — Cookies sans préfixe sécurisé

**Fichier :** `src/services/authService.js:28-29` | `app.js:70` | `src/middlewares/auth.js:13` | `src/middlewares/csrf.js:10`

**Contrainte importante :** Le préfixe `__Host-` exige que le cookie ait `Secure=true`. En développement (`secure: false`), un cookie nommé `__Host-*` est **silencieusement rejeté par le navigateur** — toute l'authentification cesserait de fonctionner en local. Utiliser `__Secure-` à la place : il exige `Secure=true` mais tolère un attribut `Domain` et n'impose pas `Path=/`, ce qui le rend compatible avec la configuration actuelle.

Renommer les cookies dans `authService.js` :
```js
// Avant :
res.cookie("accessToken", ...)
res.cookie("refreshToken", ...)
// Après :
res.cookie("__Secure-access", ...)
res.cookie("__Secure-refresh", ...)
```

Mettre à jour toutes les références :
- `auth.js:13` : `req.cookies?.accessToken` → `req.cookies?.["__Secure-access"]`
- `auth.js:44-45` : `res.clearCookie("accessToken")` → `res.clearCookie("__Secure-access")`
- `app.js:70` : `name: "__Secure-sid"`
- `csrf.js:10` : `cookieName: "__Secure-csrf"`

En développement, les cookies `__Secure-*` sont aussi rejetés par le navigateur si `secure=false`. Le plus simple est donc de nommer les cookies normalement en dev et d'appliquer le préfixe uniquement quand `isSecureContext` est vrai :

```js
// Utilisation :

```

S'assurer que `path: "/"` est explicitement défini sur chaque cookie.
